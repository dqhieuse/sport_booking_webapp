# Sport Booking WebApp - Database Design

## 1. Document Purpose

This document describes the initial database design for Sport Booking WebApp. The database uses PostgreSQL and will be mapped to Spring Boot entities using JPA/Hibernate.

## 2. Design Principles

- Each table should have a primary key using `BIGINT GENERATED BY DEFAULT AS IDENTITY`.
- Main tables should have `created_at` and `updated_at` columns.
- Important data should not be hard deleted after bookings exist; prefer status-based deactivation.
- Foreign key relationships must be clear.
- Bookings must avoid duplicates by `court_id`, `booking_date`, and `time_slot_id` for active booking statuses.

## 3. Table List

### 3.1. roles

Stores user roles.

| Column | Data type    | Note                      |
| ------ | ------------ | ------------------------- |
| id     | BIGINT       | Primary key               |
| name   | VARCHAR(50) | `USER`, `VENDOR`, `ADMIN` |

Constraints:

- `name` is unique.

### 3.2. users

Stores account information.

| Column         | Data type     | Note                                                  |
| -------------- | ------------- | ----------------------------------------------------- |
| id             | BIGINT        | Primary key                                           |
| full_name      | VARCHAR(100)  | Full name                                             |
| email          | VARCHAR(150)  | Login email                                           |
| phone          | VARCHAR(20)   | Phone number                                          |
| avatar_url     | VARCHAR(500)  | User avatar URL, nullable                            |
| password       | VARCHAR(255)  | Encrypted password, nullable for Google-only accounts |
| role_id        | BIGINT        | FK to roles                                           |
| provider       | VARCHAR(30)   | `LOCAL`, `GOOGLE`                                     |
| provider_id    | VARCHAR(255)  | External provider user id, nullable                   |
| email_verified | BOOLEAN       | Whether email is verified                             |
| status         | VARCHAR(30)   | `PENDING_VERIFICATION`, `ACTIVE`, `INACTIVE`          |
| created_at     | TIMESTAMP     | Created date                                          |
| updated_at     | TIMESTAMP     | Updated date                                          |

Constraints:

- `email` is unique.
- `role_id` references `roles(id)`.

### 3.3. email_verification_tokens

Stores email verification tokens for local email/password accounts.

| Column     | Data type    | Note                               |
| ---------- | ------------ | ---------------------------------- |
| id         | BIGINT       | Primary key                        |
| user_id    | BIGINT       | FK to users                        |
| token      | VARCHAR(255) | Verification token                 |
| expires_at | TIMESTAMP    | Token expiration time              |
| used_at    | TIMESTAMP    | Time when token was used, nullable |
| created_at | TIMESTAMP    | Created date                       |

Constraints:

- `user_id` references `users(id)`.
- `token` is unique.

### 3.4. refresh_tokens

Stores refresh tokens for login sessions. Only token hashes should be stored, not raw refresh tokens.

| Column     | Data type    | Note                                    |
| ---------- | ------------ | --------------------------------------- |
| id         | BIGINT       | Primary key                             |
| user_id    | BIGINT       | FK to users                             |
| token_hash | VARCHAR(255) | Hash of the refresh token               |
| expires_at | TIMESTAMP    | Token expiration time                   |
| revoked_at | TIMESTAMP    | Time when token was revoked, nullable   |
| created_at | TIMESTAMP    | Created date                            |

Constraints:

- `user_id` references `users(id)`.
- `token_hash` is unique.

### 3.5. sports

Stores the list of sports.

| Column      | Data type     | Note                 |
| ----------- | ------------- | -------------------- |
| id          | BIGINT        | Primary key          |
| name        | VARCHAR(100) | Sport name           |
| description | VARCHAR(500) | Description          |
| status      | VARCHAR(30)   | `ACTIVE`, `INACTIVE` |
| created_at  | TIMESTAMP     | Created date         |
| updated_at  | TIMESTAMP     | Updated date         |

Constraints:

- `name` is unique.

### 3.6. venues

Stores venue information.

| Column       | Data type      | Note                           |
| ------------ | -------------- | ------------------------------ |
| id           | BIGINT         | Primary key                    |
| vendor_id    | BIGINT         | FK to users with `VENDOR` role |
| name         | VARCHAR(150)  | Venue name                     |
| address      | VARCHAR(255)  | Address                        |
| description  | VARCHAR(1000) | Description                    |
| phone        | VARCHAR(20)    | Contact phone number           |
| opening_time | TIME           | Opening time                   |
| closing_time | TIME           | Closing time                   |
| status       | VARCHAR(30)    | `ACTIVE`, `INACTIVE`           |
| created_at   | TIMESTAMP      | Created date                   |
| updated_at   | TIMESTAMP      | Updated date                   |

Constraints:

- `vendor_id` references `users(id)`.
- A vendor can only manage venues where `venues.vendor_id` matches the current user.

### 3.7. courts

Stores each specific court.

| Column         | Data type      | Note                                |
| -------------- | -------------- | ----------------------------------- |
| id             | BIGINT         | Primary key                         |
| name           | VARCHAR(150)   | Court name                          |
| sport_id       | BIGINT         | FK to sports                        |
| venue_id       | BIGINT         | FK to venues                        |
| price_per_hour | NUMERIC(12,2)  | Hourly price                        |
| description    | VARCHAR(1000)  | Description                         |
| status         | VARCHAR(30)    | `ACTIVE`, `MAINTENANCE`, `INACTIVE` |
| created_at     | TIMESTAMP      | Created date                        |
| updated_at     | TIMESTAMP      | Updated date                        |

Constraints:

- `sport_id` references `sports(id)`.
- `venue_id` references `venues(id)`.

### 3.8. venue_images

Stores images for venues.

| Column     | Data type    | Note                         |
| ---------- | ------------ | ---------------------------- |
| id         | BIGINT       | Primary key                  |
| venue_id   | BIGINT       | FK to venues                 |
| image_url  | VARCHAR(500) | Public image URL             |
| public_id  | VARCHAR(255) | Storage provider file id     |
| sort_order | INTEGER      | Display order                |
| is_primary | BOOLEAN      | Whether this is primary image |
| created_at | TIMESTAMP    | Created date                 |

Constraints:

- `venue_id` references `venues(id)`.
- Vendor can only manage images for venues they own.

### 3.9. court_images

Stores images for courts.

| Column     | Data type    | Note                         |
| ---------- | ------------ | ---------------------------- |
| id         | BIGINT       | Primary key                  |
| court_id   | BIGINT       | FK to courts                 |
| image_url  | VARCHAR(500) | Public image URL             |
| public_id  | VARCHAR(255) | Storage provider file id     |
| sort_order | INTEGER      | Display order                |
| is_primary | BOOLEAN      | Whether this is primary image |
| created_at | TIMESTAMP    | Created date                 |

Constraints:

- `court_id` references `courts(id)`.
- Vendor can only manage images for courts under their own venues.

### 3.10. time_slots

Stores global bookable time slot templates.

| Column     | Data type   | Note                 |
| ---------- | ----------- | -------------------- |
| id         | BIGINT      | Primary key          |
| start_time | TIME        | Start time           |
| end_time   | TIME        | End time             |
| status     | VARCHAR(30) | `ACTIVE`, `INACTIVE` |

Constraints:

- `start_time` must be earlier than `end_time`.
- The pair `start_time`, `end_time` should be unique.
- `status` controls whether this global time slot template can still be used by the system.

### 3.11. court_time_slots

Stores which time slots are enabled for each court.

| Column       | Data type   | Note                 |
| ------------ | ----------- | -------------------- |
| id           | BIGINT      | Primary key          |
| court_id     | BIGINT      | FK to courts         |
| time_slot_id | BIGINT      | FK to time_slots     |
| status       | VARCHAR(30) | `ACTIVE`, `INACTIVE` |
| created_at   | TIMESTAMP   | Created date         |
| updated_at   | TIMESTAMP   | Updated date         |

Constraints:

- `court_id` references `courts(id)`.
- `time_slot_id` references `time_slots(id)`.
- The pair `court_id`, `time_slot_id` should be unique.
- A booking can only be created when the matching `court_time_slots` record is `ACTIVE`.

### 3.12. bookings

Stores court bookings.

| Column       | Data type     | Note                                                         |
| ------------ | ------------- | ------------------------------------------------------------ |
| id           | BIGINT        | Primary key                                                  |
| user_id      | BIGINT        | FK to users                                                  |
| court_id     | BIGINT        | FK to courts                                                 |
| time_slot_id | BIGINT        | FK to time_slots                                             |
| booking_date | DATE          | Booking date                                                 |
| total_price  | NUMERIC(12,2) | Total price                                                  |
| status       | VARCHAR(30)   | `PENDING`, `CONFIRMED`, `REJECTED`, `CANCELLED`, `COMPLETED` |
| note         | VARCHAR(500)  | Note                                                         |
| created_at   | TIMESTAMP     | Created date                                                 |
| updated_at   | TIMESTAMP     | Updated date                                                 |

Constraints:

- `user_id` references `users(id)`.
- `court_id` references `courts(id)`.
- `time_slot_id` references `time_slots(id)`.
- The pair `court_id`, `time_slot_id` must exist in `court_time_slots` with `ACTIVE` status when creating a booking.
- Do not allow duplicate `court_id`, `booking_date`, and `time_slot_id` when booking status is `PENDING` or `CONFIRMED`.

### 3.13. payments

Stores booking payment information. Payment status is separated from booking status to keep booking workflow simple.

| Column                  | Data type     | Note                                                                                 |
| ----------------------- | ------------- | ------------------------------------------------------------------------------------ |
| id                      | BIGINT        | Primary key                                                                          |
| booking_id              | BIGINT        | FK to bookings                                                                       |
| method                  | VARCHAR(30)   | `VNPAY`, `CASH_AT_COURT`                                                             |
| amount                  | NUMERIC(12,2) | Payment amount                                                                       |
| status                  | VARCHAR(30)   | `UNPAID`, `PENDING`, `PAID`, `FAILED`, `REFUND_PENDING`, `REFUNDED`, `REFUND_FAILED` |
| provider_transaction_id | VARCHAR(255)  | VNPAY transaction reference, nullable                                                |
| paid_at                 | TIMESTAMP     | Paid time, nullable                                                                  |
| refund_amount           | NUMERIC(12,2) | Refund amount, nullable                                                              |
| refund_reason           | VARCHAR(500)  | Refund reason, nullable                                                              |
| refunded_at             | TIMESTAMP     | Refunded time, nullable                                                              |
| created_at              | TIMESTAMP     | Created date                                                                         |
| updated_at              | TIMESTAMP     | Updated date                                                                         |

Constraints:

- `booking_id` references `bookings(id)`.
- Each booking should have one active payment record in MVP.

## 4. Table Relationships

```text
roles 1 - n users
users 1 - n bookings
users 1 - n venues
users 1 - n email_verification_tokens
users 1 - n refresh_tokens
sports 1 - n courts
venues 1 - n courts
venues 1 - n venue_images
courts 1 - n bookings
courts 1 - n court_images
time_slots 1 - n bookings
courts 1 - n court_time_slots
time_slots 1 - n court_time_slots
bookings 1 - 1 payments
```

```text
[roles]

  name
    |
    | 1-n

  email
  phone
  avatar_url
  password
  provider
  email_verified
  status
    |
    | 1-n
    v

  id PK              id PK              id PK
  user_id FK         sport_id FK        name
  court_id FK        venue_id FK        status
  time_slot_id FK    name
  booking_date       price_per_hour
  total_price        status
  status
    ^
    | n-1
[time_slots]
  id PK
  start_time
  end_time
  status

[courts] 1 - n [court_time_slots] n - 1 [time_slots]
  court_id FK
  time_slot_id FK
  status

[bookings] 1 - 1 [payments]
  booking_id FK
  method
  amount
  status
  provider_transaction_id

[users] 1 - n [email_verification_tokens]
[users] 1 - n [refresh_tokens]

[venues]
  id PK
  vendor_id FK
  name
  status
    |
    | 1-n
    v
[courts]
[users with VENDOR role] 1 - n [venues]

[venues] 1 - n [venue_images]
[courts] 1 - n [court_images]
```

## 7. Entity Implementation Notes

- Use `@Enumerated(EnumType.STRING)` for status columns.
- Create separate enums:
    - `UserStatus`
    - `CourtStatus`
    - `BookingStatus`
    - `PaymentMethod`
    - `PaymentStatus`
    - `CommonStatus`
- Use `BigDecimal` for money.
- Use `LocalDate` for `booking_date`.
- Use `LocalTime` for `start_time`, `end_time`, `opening_time`, and `closing_time`.
- Use `LocalDateTime` for `created_at` and `updated_at`.
